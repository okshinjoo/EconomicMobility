# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

"Empower" (Economic Mobility Project) is a free financial-education website aimed at first-generation, low-income, and immigrant youth. Next.js 16 (App Router) + React 19 + Tailwind CSS v4, all TypeScript. No backend yet — everything is static/client-side.

## Commands

```bash
npm run dev      # dev server at http://localhost:3000 (Turbopack)
npm run build    # production build
npm run start    # serve the production build
npm run lint     # ESLint (eslint-config-next: core-web-vitals + typescript)
```

There is no test framework configured. Verify changes by running `npm run dev` and checking the page, and `npm run build` to catch type errors (the build type-checks the whole tree).

## Architecture

Built routes: `/` (`app/page.tsx`), `/quiz`, `/about`, `/learn` (hub), `/learn/[topic]` (SSG per `TopicId`), `/learn/[topic]/[article]` (SSG, one page per article — see the Articles section for the current count), `/glossary`, `/tools` (hub), and the calculators under it. Calculators are organized into **categories** (`lib/toolsRegistry.ts`): each category has a *main* calculator at `/tools/{budget,debt,savings,college}` plus focused *sub*-calculators nested under it (e.g. `/tools/debt/auto-loan`, `/tools/debt/mortgage`, `/tools/budget/paycheck`, `/tools/savings/compound`, `/tools/savings/retirement`, `/tools/savings/roth-ira`, `/tools/college/student-loan`). Every calc page shows `components/CalcSwitcher.tsx` (pill nav across the category; "soon" items in the registry render disabled). New sub-calc pages use `components/CalcPageShell.tsx`. To add a calculator: add it to `toolsRegistry.ts`, then build its page (it appears in the switcher + hub automatically). Invalid `[topic]`/`[article]` slugs `notFound()`. Also built: `/start-here` (onboarding + FAQ), `/resources` (curated external links + a per-state finder, see below), `/contact` (`components/ContactForm.tsx` — sends in-page via Web3Forms to `Help@economicmobilityproject.org` with an inline confirmation; `mailto` kept only as a fallback link. Same access key as the Ask/Community forms), `/tools/templates` (downloadable spreadsheet templates, see below), and `/ask` (the **"Ask Empower" Q&A** — anonymous question form + curated answered questions, see below). **Every header/footer nav link now resolves** — no 404s.

**Reality Check** (`/tools/budget/reality-check`): the Jump$tart-style "pick the life you want, see the salary it takes" stepper. Data + solver in `lib/realityCheck.ts` (categories/options with national-ballpark monthly costs; `salaryForLifestyle()` binary-searches `estimateTaxes` so the result is a real GROSS salary for a single filer in the chosen state, not a flat "add 22%"), UI in `components/RealityCheckTool.tsx` (Jump\$tart-style GROUPED slides - 15 categories over 5 screens via the `SLIDES` map; the four choice-based slides come first and ALL five type-it-in (`freeEntry`) categories are grouped on the final 'Your own numbers' slide beside the state picker (owner request: manual input only on the last page); prices are deliberately HIDDEN during selection so picks aren't anchored; fun/travel/savings/debts/family are `freeEntry` categories (typed MoneyInput on the slide, stored in `overrides`, because amounts vary too widely for presets); student loans are their own none/average/more-than-average choice (Jump\$tart style), separate from the fill-in debts category; snapshots from before the debt split (a `debt` pick) start fresh instead of resuming; the Home slide has a required big-city-vs-typical toggle (`col`) that switches housing options to their `monthlyHigh` prices; results show the salary as a \$5,000 band plus a 'How we priced your picks' receipt where every category's estimate is a MoneyInput the user can correct - corrections live in `overrides` in the snapshot and recompute the total/band live; snapshot `{picks, stateCode, overrides}` saved to `STORAGE_KEYS.realityCheck` and resumed). Update the option costs occasionally like article figures.

**Downloadable templates** (`/tools/templates`): three `.xlsx` files with live SUM/IFERROR formulas + brand styling (budget, debt payoff tracker, savings goal worksheet) PLUS three career `.docx` files (first resume / student resume / cover letter, July 16 2026 — teaching placeholders in [brackets], one page, US Letter). **Courses** (`/courses`, `/courses/[course]`): focused learning modules in `lib/courses.ts` — each is a curated cross-topic reading path (`articleSlugs`), an auto-derived flashcard deck, and a hand-written 6-question final (`finalQuiz`, each question carries a `sourceSlug` so wrong answers link "Review: <article>"). Flashcards are NEVER hand-written: `getCourseFlashcards()` scans the course's articles with the glossary matcher, so decks stay in sync — to enrich a thin deck, add terms to `lib/glossary.ts` (they also start auto-linking site-wide). The final is gated client-side on all module articles being read (read tracking), with a small "take it anyway" escape hatch; passing (≥`PASS_PERCENT` 80%) awards a local-only badge (`empower:course-badges:v1`, rosette SVG `BadgeMedal` in `components/CourseQuiz.tsx`). Client pieces: `CourseGrid` (hub cards with live progress), `Flashcards` (flip deck), `CourseQuiz`. Adding a course = one entry in `lib/courses.ts` (final-quiz questions must not duplicate the article-level `quiz` stems). Nav entry "Courses" in `lib/nav.ts`. **Draft courses (July 13, 2026, owner ask "show that it's still in progress"):** `draft: true` + `finalQuiz: []` ships a course with its reading path + auto flashcards live but an honest in-progress state everywhere — hub card swaps the medal for an "In progress" chip and adjusts its meta/progress copy, the course page adds the chip to the hero kicker and renders an in-progress card where the final would sit (no badge earnable until the final exists — badges only ever come from passed finals). ### The quiz (the only real logic in the app)

The quiz is the centerpiece. Its full product spec lives in `docs/quiz-spec.md` — **read it before changing quiz behavior, and update it in the same change**, since it's the source of truth for question wording, scoring rules, and results copy.

Data and logic are fully separated from rendering:

- **`lib/topics.ts`** — the 9 canonical financial topics (`TopicId` union, icons, colors, `/learn/*` hrefs). Adding a `TopicId` is a spine change: TypeScript forces matching entries in `learnContent`, and in `quizData.ts`'s `knowledgeCheckBank` / `topicWhyBlurbs` / `topicResourceCards` records (plus an optional Q3 option). Everything driven by `topics.map()` — homepage cards, Learn hub, nav dropdowns, static params — updates automatically. `getTopic(id)` throws on unknown ids. This list is the spine: homepage topic cards, quiz topics, knowledge-check banks, and resource cards all key off `TopicId`.
- **`lib/quizData.ts`** — all quiz content + pure scoring/results functions: question definitions (`quizQuestions`), the knowledge-check question bank (`knowledgeCheckBank`, keyed by `TopicId` → `Tier`), and result generators (`getFinancialProfile`, `getKnowledgeCheckFeedback`, `getWhereToStart`, `scoreKnowledgeCheck`, resource-card maps). Keep this file logic-only and presentation-free.
- **`components/QuizFlow.tsx`** (`"use client"`) — the state machine: `welcome → q1..q5 → kc → results`. Holds all answer state and drives navigation/progress. The knowledge check (`kc`) is built dynamically from Q3-selected topics at the difficulty `Tier` derived from Q5 (`confidenceTier`), pulling `KC_QUESTIONS_PER_TOPIC` (currently 2) questions per topic.
- **`components/QuizResults.tsx`** — pure presentation; takes computed answers/tier and calls the `lib/quizData.ts` functions to render the 4-part results page.

Key branching rules baked into the flow (see spec for rationale):
- **`not-sure`** on Q3 is **combinable** with real topics (plain toggle, not exclusive). If real topics are also chosen, the knowledge check runs on those topics as normal. If *only* `not-sure` is chosen, a short **general knowledge check** runs instead (`GENERAL_KC_PICKS` in `quizData.ts` — one beginner question each from a few foundational topics) and results use "let's figure it out together" framing with all topics for guidance. The knowledge check (topic or general) is also **skippable** mid-flow via a footer ("Skip this question" / "Skip to my results"); skipping sets `skippedKc`, which suppresses score feedback on the results page.
- Q5 confidence maps to a `Tier` (`beginner`/`advanced`) that selects which knowledge-check questions are asked *and* which resource-card variant is shown.
- The **confidence-mismatch** feedback messages (said-confident-but-scored-low, said-beginner-but-scored-well) are intentional and should not be removed — they're what makes the quiz feel honest.
